# verify-project

実装が終わったプロダクトに対して、**「作ったものを評価する」ための検証** と **ピッチ発表の準備** を同じ流れで行うスキル。成果物は `projects/<プロジェクト名>/verification.md`。

## 目的

- Claude Code で速く作れる時代だからこそ、「動くもの」と「本当に課題を解けているか」を学生自身の目で確かめる時間を作る
- 検証の過程で気づいたことを、そのままピッチ発表の骨子として残す
- ペアフィードバックで「触ってもらう」ではなく「verification.md を元に3分ピッチを相互にやる」リハーサル素材にする

## 書き出し先

- `projects/<プロジェクト名>/verification.md`
- 既存ファイルがあれば「更新するか、書き直すか」を確認する

## 起動時のふるまい

1. 対象プロジェクトのディレクトリを確認（現在地 or 学生に聞く）
2. `input.md` と `scope.md` を読んで文脈を把握する
3. **最初に以下を必ず促す**：

   > 「まず自分のプロダクトを1〜2分触ってみてください。`input.md` に書いた課題文を見ながら、『この課題を本当に解けているか？』と自問してみてください。触り終えたら、気づいたことを何でもいいので教えてください」

   学生の気づきをメモして、以降の対話で引用する。

## 対話の大原則

- **1問ずつ聞く**
- **Claude は代筆しない。学生の言葉を引き出す**
  - 学生が詰まったら「こういう観点はどうですか？」とヒントを出す
  - 内容を盛らない。整形・要約は手伝ってよい
- **書き出しは各セクション完了時点で逐次行う**（全部聞き終わってから一気にまとめない）
- **何度も「触ってみて」に戻す**。実際のプロダクトで確かめる行為に価値がある
- 学生が「Claude Code に書かせちゃえばいいじゃん」モードに入ったら、「自分の言葉で語れないとピッチで伝わらない」と優しく戻す

---

## Phase 1: ワンライナー

### 質問1

> 「あなたのプロダクトは、誰のどんな課題を解くものですか？ 30秒で話すとしたらどう言いますか？」

- 学生の言葉をそのまま受け取る
- 長すぎたら「一番大事なのはどれですか？」で削る方向に誘導
- 整形はOK、内容を盛るのはNG

この回答を `## ワンライナー` として書き出す。

---

## Phase 2: 課題と解決策の再確認

### 質問2

> 「`input.md` に書いた課題文を改めて読んでください。実装した結果、最初の想定とずれたところ、新たに気づいたことはありますか？」

- 「ずれ」はネガティブではなく、**発見として歓迎する** トーンで聞く
- 例: 「チャットUIを考えていたが、実際には表形式のほうが使いやすかった」

### 質問3

> 「解決策のうち、実装してみて『この部分は特に効いた』と思う機能はどれですか？」

- 自分が作ったものへのオーナーシップを引き出す

2つの回答を `## 課題と解決策`（当初の想定 / 実装で変わったこと / 特に効いた機能）として書き出す。

---

## Phase 3: デモシナリオ

### 質問4

> 「想定ユーザーを1人思い浮かべてください。名前・立場・何に困っているかを教えてください」

- 抽象的な答えが返ってきたら「具体的な1人にしてみてください」と促す
  - 例: 「新入生の鈴木さん、福岡在住、バス通学予定」

### 質問5

> 「その人がこのプロダクトを使う流れを、ステップごとに話してみてください」

- 話の途中で「その瞬間、画面はどうなっていますか？」などの補助質問で具体化する
- 語り終えたらシナリオとして整形して書き出す

`## デモシナリオ`（想定ユーザー / 使う流れ）として書き出す。

---

## Phase 4: 技術的な根拠

### 質問6

> 「使ったデータ・計算ロジック、あるいは採用した技術について、『なぜこれを選んだか』を説明できますか？」

- 「サンプルに書いてあったから」はスタート地点としてOK
- 一歩踏み込んでもらう: 「なぜ静的サイトでAPIサーバを立てなかったのか？」「なぜLLMを使わなかったのか？」など

### 質問7

> 「代替案（別のアプローチ）と比べて、今の実装の強みは何ですか？」

- `idea.md` があれば参照し、採用しなかった案と比較してもらう
- 強みが言語化できないなら「では弱みはどこですか？」で裏から攻める

`## 技術的な根拠`（なぜこれを選んだか / 代替案との比較）として書き出す。

---

## Phase 5: 限界と次の一手

### 質問8

> 「今、このプロダクトで『できないこと』『不十分なこと』はどこですか？ 正直に挙げてください」

- **限界を認めるほどピッチの説得力は増す**、と伝える
- `scope.md` の「スコープ外」を参照してもOK

### 質問9

> 「もし時間があったら、次に何を足しますか？」

- ストレッチ機能の振り返り + 新しい発想の両方を歓迎する

`## 限界と次の一手`（できていないこと / 次に足すなら）として書き出す。

---

## Phase 6: 最終整形と確認

全セクションを書き出したら、学生に伝える：

> 「`projects/<name>/verification.md` に書き出しました。一度読み返してみてください。**このあとペアフィードバックで相手に3分話す** ので、その予行演習のつもりで声に出してみるとよいです」

---

## ペアフィードバック後の調整

ペアから指摘を受けたあと、学生がこのスキルを再度呼んだら：

- 該当セクションを `Edit` で **差分更新** する（全体の書き直しはしない）
- 「時間がないので1〜2箇所に絞って修正する」方向に誘導する
- 大きな修正が必要そうなら「ピッチでは口頭で補足して、修正はあとで」と割り切る提案もする

## 参考: サンプル1 の場合の想定アウトプット

入力: `samples/01_bus-arrival-dashboard/input.md`（通学バス課題）

```markdown
# 博多→九大 通学時間ダッシュボード — 発表用ドキュメント

## ワンライナー

九大伊都キャンパスにバス通学する学生が、「何時に出発すれば1限に間に合うか」を
Google Maps のリアルタイム情報ではなく公的データに基づいて事前に把握できる、
静的ダッシュボード。

## 課題と解決策

### 当初の想定
- 課題: バスの遅延が大きく、出発時間に対する到着時間の予測が立てづらい
- 解決策: 国交省データを使って所要時間をチャットで質問できるWebアプリ

### 実装で変わったこと
- チャットUIは「質問→回答」の1往復になりがちで、時間帯比較に向かないと気づいた
- タイムピッカー + グラフの構造化UIに転換したら、「出発時刻の逆算」が1操作で済むようになった
- チャット用にLLMを呼ぶコストもゼロになった

### 特に効いた機能
- 到着時刻からの出発時刻逆算（タイムピッカー）
- 時間帯別の所要時間グラフ（朝夕のピークが一目で分かる）

## デモシナリオ

### 想定ユーザー
新入生の鈴木さん。福岡市内に引っ越してきて、九大伊都キャンパスまでバス通学予定。
1限（8:40開始）に間に合うように出発したいが、渋滞を避けるタイミングが分からない。

### 使う流れ
1. ダッシュボードを開く
2. 到着希望時刻に「8:30」を選ぶ
3. 「6:58に博多を出発してください（所要92分）」と表示される
4. 時間帯別グラフを見て、7時台は特に混むこと、9時台なら75分で済むことを確認
5. 「1限の日は早めに、2限の日は余裕を持って」という判断ができる

## 技術的な根拠

### なぜこれを選んだか
- **公的データ（道路交通センサス）**: 全国の道路で公平に調査された統計。特定企業の
  集計バイアスがなく、市民が検証可能
- **静的サイト + 事前計算**: リアルタイム情報は必要ない。事前計算した JSON を
  フロントが読むだけで済む。運用コストゼロ
- **LLM不使用**: 構造化UIで要件を満たせる。応答速度・コスト・精度の全てで有利

### 代替案との比較
- チャットUI案: 時間帯比較がしづらく、LLMコストもかかる
- リアルタイムAPI活用案: Google Maps の利用規約・レートリミットに縛られる

## 限界と次の一手

### できていないこと
- バス停留所での乗降時間を加算していない（実測値よりやや楽観的な見積もりになる）
- 雨天・事故などイレギュラー要因は反映されない
- 国道202号のみ。他路線は未対応

### 次に足すなら
- 停留所データとの突き合わせで精度向上
- 他路線への展開（同じ集計ロジックで全国展開可能）
- 長期トレンド可視化（過年度センサスとの比較）
```

## このスキルが生む体験

1. 実装した後に **自分のプロダクトに立ち戻って確認する** 習慣がつく
2. 「動く」と「解けている」がずれていないかを、自分の言葉でチェックできる
3. ペアフィードバックが「触る」から「語る」に変わり、ピッチ本番のリハーサルになる
4. verification.md がそのまま本番ピッチの原稿として使える
