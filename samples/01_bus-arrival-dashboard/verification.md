# 博多→九大 バス通学時間ダッシュボード — 発表用ドキュメント

## ワンライナー

九大へ通う学生の、「授業に間に合わせたいので道路混雑の影響で何時に出発すればいいかわからない」という課題を解くものです。

## 課題と解決策

### 当初の想定

- **課題**: 九州大学への通学時にバスの遅延が大きく、自宅を出発する時間に対する学校の到着時間の予測が立てづらい
- **解決策**: 国土交通省のデータを活用して、博多から九大に行くバスがおおよそどれくらいの時間を要するかチャットで質問できるWebアプリを作る

### 実装で変わったこと・気づいたこと

- **チャットUIではなくグラフ中心のダッシュボードに着地**: 実際にピンポイントで時刻を聞くより、**周辺時間帯の混雑具合も帯で見えたほうが予測が立てやすい**。ピーク時と平常時の差分が一目で分かることで「だいたいこれくらい混むんだな」という肌感覚が得られる
- **出発エリアの選択肢が意外と狭い**: 今は国道202号沿いの 10 地点のみ。不特定多数に使ってもらうなら、もう少し広いエリアカバレッジが必要
- **データの鮮度**: 使っている道路交通センサスは令和3年度（2021年）データ。予測の参考値としては十分だが、最新性という観点では物足りない

### 特に効いた機能

- **速度データ → 所要時間(分) への変換**: 元データは「各区間の旅行速度（km/h）」だが、ユーザーが行動判断に使えるのは「所要時間（分）」。距離と速度から時間を計算してユーザー向け単位に揃えたことで、**「授業に間に合うか」を直接判断できる**出力になっている

## デモシナリオ

### 想定ユーザー

工学部2年生。1時限目に必須の実験があるため、伊都キャンパスに遅刻できない。博多エリア在住でバス通学。

### 使う流れ（前夜ルーティン）

1. 実験前夜、**寝る前にスマホで** ダッシュボードを開く（翌朝の起床時刻を逆算したい）
2. 出発地「博多駅周辺」、到着希望時刻「8:30」をセット
3. 画面上部の **「7:00頃に博多を出発 ／ 所要時間 約90分」** を最初に確認
4. 逆算で「起床は 6:30、準備 30分」と決めて目覚ましをセット
5. 時間帯別グラフも覗いて、もし寝坊してピーク帯に入ると +20分 ずれる、と頭に入れておく
6. 就寝

## 技術的な根拠

### なぜこれを選んだか

- **公的データ（道路交通センサス）を採用**: 説得力のあるオープンデータが欲しかった。特定企業の集計ではなく、誰もが検証・再利用できる客観的なソース
- **国交省のオープンデータ基盤の存在**: 自治体データがここまで整備されて公開されている、という発見そのものが採用の決め手。同じ手法が全国の道路に展開できる
- **Next.js版も並置**: 静的サイト版（HTML単体）だけでなく Next.js 版（API層を挟む構成）も作って、「サーバーサイドを入れると役割分担がどう変わるか」を自分で体感できるようにした（学習目的）

### 代替案との比較（案A: LLMチャット型 との比較）

- **強みは"軽さ"と"即応性"**: 最高の正確さよりも「早く見たい」シチュエーションが多い（前夜に寝る前に確認、朝の忙しい時間にサッと確認、など）。LLM を挟むと応答に数秒〜十秒かかるが、静的ダッシュボードは即座に答えが出る
- **運用コストゼロ**: LLM API コストがかからない。教育機関・自治体が低コストで展開できる

## 限界と次の一手

### できていないこと

- **ユーザー設定の永続化がない**: 自分の出発地はほぼ毎回同じなのに、ページを開き直すたびに選び直す必要がある。localStorage 等で設定を覚えておく仕組みが未実装。使用頻度が高いユーザーほど不満になる
- **データの鮮度**: 令和3年度（2021年）の道路交通センサスを使用。直近のトレンド変化は反映されていない
- **出発地の選択肢が狭い**: 国道202号沿いの 10 地点のみ。不特定多数向けには地点カバレッジが不足

### 次に足すなら

scope.md のストレッチに書いていた 3 つをまず完走したい：

1. **CSV → SQLite 化**: 毎回の CSV パースをやめて、DB からクエリで必要な部分だけ取り出す構成に切り替える。将来の拡張（複数路線・区間ごとの個別参照）の基礎にもなる
2. **上り／下り切替**: 通学（博多→九大）だけでなく帰宅（九大→博多）方向の所要時間も見られるようにする
3. **区間別ヒートマップ**: 区間 × 時間帯のヒートマップで、どの区間が一番のボトルネックかを一目で示す
