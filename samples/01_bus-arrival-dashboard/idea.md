# 実装アプローチ案

入力: `input.md`

- 課題: 九州大学への通学時にバスの遅延が大きく、自宅を出発する時間に対する学校の到着時間の予測が立てづらい
- 解決策: 国土交通省のデータを活用して、博多から九大に行くバスがおおよそどれくらいの時間を要するかチャットで質問できるWebアプリを作る

入力の解決策は「チャット型Webアプリ」という具体的な UI を含んでいるが、**それ以外の実装形式も含めて** アプローチを発散させる。

---

## 案A: チャット質問型 Webアプリ（LLM 利用）

- **何をするか**: 「博多から10時に着きたい、何時に出ればいい？」のように自然言語で質問するとLLMが交通量データを参照して応答する Webアプリ
- **できる**:
  - 自由度の高い質問（「朝ピーク避けたら何時がいい？」等）
  - 対話の流れでの条件変更
- **できない**:
  - 応答速度がLLM API依存で遅め
  - 推論の揺れ（毎回同じ結果になるとは限らない）
  - API利用料がかかる
- **難易度**: 高い。LLMにデータを渡すための前処理、プロンプト設計、API連携。180分でフル実装は厳しく、中途半端になるリスク
- **学びどころ**: LLM×構造化データ（RAG的パターン）、Tool Use の考え方、自然言語インターフェースの設計

---

## 案B: 静的ダッシュボード Web サイト（LLM 不使用）

- **何をするか**: CSV から事前計算した所要時間を JSON に書き出し、静的 HTML が読み込んでグラフ・表・タイムピッカーで可視化する
- **できる**:
  - 即時応答（API呼び出しなし）
  - サーバー不要、無料ホスティング
  - オフラインでも動く
  - 何度動かしても同じ結果（再現性◎）
- **できない**:
  - 自由な自然言語質問は受け付けない（構造化UIで代替）
  - インタラクションは事前に設計した範囲のみ
- **難易度**: 中。Pythonで CSV 集計 + Chart.js で描画。180分で十分に形になる
- **学びどころ**: 公的データの事前集計、「重い処理はビルド時に・軽いデータをフロントへ」というデータジャーナリズムの定石、静的サイトの威力

---

## 案C: CLI ツール（LLM 不使用・ターミナル）

- **何をするか**: `bus-time hakata --arrive 10:00` のようにコマンドを実行すると、ターミナルに推奨出発時刻と時間帯別所要時間の表／アスキーグラフが出る
- **できる**:
  - 起動が一瞬、ターミナルだけで完結
  - シェルスクリプトに組み込みやすい
  - 表示はRichでそれなりに綺麗になる
- **できない**:
  - 非エンジニアに配布しづらい（インストール手順が必要）
  - 見栄えはWebに劣る（プレゼン映えは弱め）
- **難易度**: 低〜中。Python + Click + Rich のシンプル構成。180分で余裕を持って作れる
- **学びどころ**: CLI 設計、ターミナル上の表・グラフ描画、UNIXフィルタ的な思想（入力→整形→出力）
