# 会話ログ: 博多→九大 通学時間ダッシュボード

このファイルは、サンプルプロジェクト `01_bus-arrival-dashboard` を Claude Code で作るまでに、わたしと Claude がどうやりとりしたかのダイジェストです。

完全な再現ではなく、「こういう聞き方をすると、こう返ってきて、こう絞り込んでいった」という流れを追えるようにしています。

## 1. 最初の相談（課題と解決策の持ち込み）

### わたし

> こういう課題と解決策で考えてます。まず input.md を整理したいので、この内容を読み込んで、そのまま書き出してください。
>
> - 課題: 九州大学への通学時にバスの遅延が大きく、自宅を出発する時間に対する学校の到着時間の予測が立てづらい
> - 解決策: 国土交通省のデータを活用して、博多から九大に行くバスがおおよそどれくらいの時間を要するかチャットで質問できるWebアプリを作る

### Claude

`input.md` に書き出しました。気になっている点として、この解決策には2つの要素が混ざっています。

1. **データの活用**: 国交省のデータで所要時間を推定する
2. **UIの形**: チャット形式で質問できる Web アプリにする

この2つは独立して議論できます。特に（2）については「本当にチャット形式が最適か？」を一度検討してみる価値があります。

**質問:** 学生が知りたいのは「具体的に何時に出発すればいいか」でしょうか？それとも「いろんな時間帯の所要時間を自由に聞きたい」でしょうか？

## 2. アプローチの比較検討

### わたし

> 前者に近いです。1限に間に合わせたい、みたいな具体的なケースが多い。チャットじゃないほうがいいってこと？

### Claude

「具体的なケース」が中心なら、**チャットよりも構造化UIのほうが向いている**可能性が高いです。比較してみます。

| 観点 | チャットUI（LLM経由） | 構造化UI（ダッシュボード） |
|---|---|---|
| 体験 | 自由に聞ける | 操作が早い、迷わない |
| 精度 | LLMの解釈誤りが起こりうる | 入力が限定されているのでブレない |
| 実装コスト | LLM API 連携・プロンプト調整が必要 | 静的サイトで済む |
| 運用コスト | API 課金 | ゼロ |
| ハッカソン的な強み | AI感はあるが本質と無関係になりがち | 「公的データを可視化した」という明確な価値 |

学生が本当に欲しいのが「8:32に出発すれば10:00に着きます」という情報なら、タイムピッカー1つとグラフがあれば事足ります。LLMを挟むと、むしろ不確実性が増えます。

**提案:** 到着希望時刻をタイムピッカーで入れると、出発時刻と時間帯別の所要時間グラフが出る、という静的ダッシュボードにする。

## 3. データの確認

### わたし

> 方針OK。じゃあ国交省のデータを先に見て、何ができそうか一緒に考えてほしい。data/ に kasyo40.csv と zkntrf40.csv を置いた。

### Claude

読み込んでみます。まず確認したいこと:

- CSVは **Shift_JIS** エンコーディングでした。UTF-8で読むと文字化けします
- カラム名に全角数字・全角括弧が混ざっています

中身を整理すると、2つのCSVは以下の関係です。

```
kasyo40.csv（箇所データ）       ← 区間の属性: 路線名、延長、混雑時/非混雑時の旅行速度
   ↕ JOIN キー: 都道府県指定市コード + 調査単位区間番号
zkntrf40.csv（交通量データ）    ← 時間帯別の交通量（7時台〜翌6時台）
```

**わかる情報:**

- 国道202号（博多→糸島）の区間ごとに、混雑時と非混雑時の旅行速度が記録されている
- 時間帯別の交通量があるので、「どの時間帯がどれくらい混雑しているか」を定量化できる

**わからないこと:**

- リアルタイム情報はない（2021年度の調査データ）
- バスの停留所の乗降時間は含まれない

**提案:** 時間帯別の交通量から「混雑度」を算出し、混雑時と非混雑時の速度を**線形補間**すれば、時間帯別の推定速度が出せます。これを区間ごとに積み上げれば、時間帯別の所要時間になります。

## 4. スコープの絞り込み

### わたし

> いいね。この方針で scope.md を書いてほしい。MVPとストレッチは分けて。

### Claude

`scope.md` を作成しました。要点:

**MVP（必ず作る）**
1. 時間帯別所要時間グラフ（折れ線）
2. 区間別旅行速度テーブル
3. 出発時刻シミュレーター（到着希望時刻 → 出発時刻の逆算）

**ストレッチ（時間があれば）**
4. 区間別速度ヒートマップ
5. 上り/下り切替
6. 時間帯別交通量グラフ

**やらないこと:**

- リアルタイム交通情報
- 地図上の経路表示
- バス停の乗降時間
- LLM応答
- ユーザー認証

**質問:** 技術スタックですが、「ビルド不要・code-serverで即動く」を優先すると、素の HTML + JS + Chart.js (CDN) が最速です。Next.js などのフレームワークを使う必要は今回なさそうですが、どうしますか？

## 5. 技術スタックの決定

### わたし

> 素のHTML+JSでOK。データ加工はPythonで事前にやる感じ？

### Claude

はい、その構成が一番シンプルです。

```
[国交省CSV] → Python スクリプト（ビルド時に1回実行） → JSON → 静的HTML が読み込む
```

「重い計算は事前に、軽いデータをフロントへ」というデータジャーナリズムの基本パターンです。

**メリット:**
- ランニングコストゼロ
- オフラインでも動く
- CSV パース・集計ロジックが Python で簡潔に書ける

**デメリット:**
- データ更新にはビルドの再実行が必要（年1回のセンサス更新なので問題なし）

JSON の出力スキーマをまず決めておくと、Python と JS を並行で実装できます。scope.md にスキーマを書いておきました。

## 7. 実装に入る

### わたし

> モックOK。じゃあ実装進めて。まず Python 側（build_route_data.py）から。

### Claude

承知しました。scope.md の「データ加工の方針」に沿って進めます。

1. CSV 読み込み（Shift_JIS、全角カラム対応）
2. 国道202号の区間抽出（博多→糸島方向に並び替え）
3. 時間帯別交通量の集計（上下・大小型車を合算）
4. 混雑度 → 推定速度の線形補間
5. 区間ごとの所要時間を積み上げ → JSON 出力

途中、区間番号の対応付けで同一観測地点に複数行ある（区間分割）のを見落としそうになりましたが、scope.md の「注意点」に書いておいたので補正できました。

## 振り返り: このプロジェクトで効いたプロンプトのパターン

1. **課題と解決策を「分ける」**
   最初の解決策にはデータ活用とチャットUIが混ざっていた。分けることで「チャットでなくていい」に気づけた

2. **「これで合ってる？」を Claude に聞く**
   「チャットじゃないほうがいいってこと？」と投げると、比較表を作って根拠を示してくれた

3. **実データを先に見る**
   CSV を渡してから設計に入ったので、「Shift_JIS」「全角カラム」「区間分割」のような実装上の落とし穴を事前に洗い出せた

4. **スコープを文書化してから実装に入る**
   input.md → scope.md → mock.html → 実装、の順で作ったので「やらないこと」が明確になり、実装中の寄り道が減った

5. **「やらないこと」を明示する**
   リアルタイム情報・地図・LLM・認証、を最初に「スコープ外」と書いたことで、実装中に迷う場面が激減した

## 学生向けメモ

- **最初のプロンプトは「課題 + 解決策」の2行でOK**。そこから Claude と対話して絞り込んでいくのが自然
- **「AIっぽい機能を入れなきゃ」と思い込まない**。LLM を挟むのが最適解でないことも多い。今回も LLM を使わないほうが精度・コスト・体験のすべてで優れていた
- **実装前に scope.md とモックを作る**。設計に時間をかけたほうが、実装そのものは Claude Code が一気に片付けてくれる
