# スコープ設計書: 博多→九大 バス通学時間ダッシュボード

## 1. 概要

| 項目 | 内容 |
|---|---|
| 課題 | 九州大学への通学時にバスの遅延が大きく、自宅を出発する時間に対する学校の到着時間の予測が立てづらい |
| 解決策 | 国土交通省の道路交通センサスデータを活用して、博多から九大に行くバスがおおよそどれくらいの時間を要するかを、時間帯ごとにブラウザで可視化するダッシュボードを作る |
| 対象ユーザー | 博多エリアから九州大学伊都キャンパスに通学する学生 |
| アーキテクチャ | CSV から事前計算した JSON を静的サイトが読み込む構成。バックエンド不要 |

## 2. 使用データ

国土交通省「全国道路・街路交通情勢調査（道路交通センサス）」の福岡県データを使用する。

### データファイル

| ファイル | 内容 | 主な用途 |
|---|---|---|
| kasyo40.csv | 箇所データ（道路区間の属性情報） | 路線名、区間延長（km）、旅行速度（混雑時/非混雑時） |
| zkntrf40.csv | 交通量データ（時間帯別） | 各区間の時間帯別交通量（7時台〜翌6時台） |

### 2つのCSVの関係

```
kasyo40.csv（箇所データ）
  交通量／都道府県指定市コード + 交通量／調査単位区間番号
    ↕ JOIN
zkntrf40.csv（交通量データ）
  都道府県指定市コード + 交通量調査単位区間番号
```

### 対象路線: 一般国道202号

博多区 → 中央区 → 城南区 → 早良区 → 西区 → 糸島市を通り、九大伊都キャンパス方面へ至る主要路線。

### 注意点

- CSVはShift_JIS エンコーディング。読み込み時に変換が必要
- カラム名に全角数字・全角括弧が含まれる
- 車種区分: 1=小型車, 2=大型車。バスは一般交通と同じ車線を走るため、車種を問わず一般交通の旅行速度で推定する
- kasyo40には同一観測地点で複数行があるため、経路全体の所要時間は各区間を積み上げて計算

## 3. 機能一覧

### MVP（必ず作る）

| # | 機能 | 説明 |
|---|---|---|
| 1 | 時間帯ごとの所要時間可視化 | 横軸=時間帯（7〜22時）、縦軸=推定所要時間の折れ線グラフ。朝夕ラッシュの所要時間増加が一目で分かる |
| 2 | バス通学所要時間シミュレーター | 博多周辺からバス通学すると仮定した所要時間を、出発地と到着希望時刻から逆算して表示 |
| 3 | ブラウザで動作 | 静的サイトとしてブラウザで起動し、ページ単体で UI が成立 |

### ストレッチ（時間があれば）

| # | 機能 | 説明 |
|---|---|---|
| 4 | CSV → SQLite 化 | 集計済みデータを SQLite に取り込み、クエリで必要な部分を取り出す構成に切り替え |
| 5 | 上り/下り切替 | 通学（博多→九大）と帰宅（九大→博多）を切り替えて表示 |

## 7. 実装ロジック

CSV からの集計はビルド時に1回だけ実行し、結果を JSON として出力する。フロントエンドはこの JSON のみを参照する。

- **ステップ1**: 対象区間の抽出 — kasyo40.csv から一般国道202号を抽出し、博多→糸島方向に並べる
- **ステップ2**: 時間帯別交通量の取得 — zkntrf40.csv から対象区間の交通量を時間帯（7時〜翌6時）ごとに集計
- **ステップ3**: 混雑度の算出
  ```
  混雑度(h) = その時間帯の交通量 / ピーク時間帯の交通量
  ```
- **ステップ4**: 時間帯別の推定旅行速度 — 混雑時・非混雑時の旅行速度を混雑度で線形補間
  ```
  推定速度(h) = 非混雑時速度 × (1 - 混雑度(h)) + 混雑時速度 × 混雑度(h)
  ```
- **ステップ5**: 時間帯別の所要時間算出 — 各区間の所要時間を積み上げ
  ```
  所要時間(h) = Σ 区間延長(km) / 推定速度(h, 区間) × 60  （分）
  ```

## 8. 実装計画（2コマ = 180分）

Claude Code 前提。「一気に生成 → 動かして対話でチューニング」のサイクルで進める。

### 1コマ目（90分）: MVP が動く状態まで

- **初期一括生成（0〜15分）**: scope.md を Claude Code に渡して、CSV 集計スクリプト + 静的 HTML + Chart.js のドラフトを一気に生成
- **動作確認（15〜30分）**: ブラウザで開いて挙動確認。致命的な不具合の修正
- **対話チューニング（30〜90分）**: グラフの見た目、シミュレーターのロジック、UI の粒度を対話で詰める（1サイクル 10〜20分 × 3〜4本）

**目標**: 1コマ目終了時点で MVP 3機能（グラフ・シミュレーター・ブラウザ UI）が動いている

### 2コマ目（90分）: 仕上げ + ストレッチ

- **MVP 仕上げ（0〜30分）**: UI の詰め（コントラスト・レスポンシブ・見出し・空状態）を対話で
- **ストレッチ（30〜90分）**: 対話サイクル単位で順に積む。優先順は **① CSV → SQLite 化 → ② 上り/下り切替 → ③ 区間別ヒートマップ**。1つあたり 15〜25分目安

## 9. スコープ外（やらないこと）

締切を守るために、以下は今回**あえて扱わない**と宣言する：

- **LLM を使わない** — 自然言語質問・自動要約コメントなどは対象外。構造化UIで完結させる
- **認証なし** — ユーザー登録・ログイン機能は扱わない。公開前提
- **地図表示なし** — Leaflet / Mapbox 等の地図ライブラリは使わない。グラフと表のみで可視化
